## 2026-03-02: save-progress 스킬 개발

### 1. 스킬 생성은 간단하게

**문제**:
- skill-creator 가이드를 너무 충실히 따름
- 서브에이전트 6개, 벤치마크 전체 세트 실행
- 시간과 리소스 낭비

**교훈**:
- ✅ 간단한 스킬은 작성 → 직접 테스트 → 수정으로 충분
- ✅ 복잡한 테스트는 큰 스킬에만 적용
- ✅ 규모에 맞게 접근하기

---

### 2. 요구사항을 정확히 파악하기

**문제**:
- 사용자: "Git 저장, progress.md 업데이트, 글로벌 교훈 저장"
- 구현: README.md, work-summary.txt 등 불필요한 파일 생성
- 원인: 구현 전 요구사항 확인 생략

**교훈**:
- ✅ "정확히 어떤 파일을 원하시나요?" 질문하기
- ✅ 불필요한 것 명시 → "README 같은 건 만들지 말아주세요"
- ✅ 작은 프로토타입으로 먼저 검증

---

### 3. 글로벌 교훈 중복/중요도 체크

**배경**:
- 글로벌 CLAUDE.md에 교훈을 자동으로 추가하는 기능
- 중복되거나 중요하지 않은 내용까지 추가될 위험

**해결**:
1. **중복 체크**: 맥락/의도 기반 비교
   - 키워드 매칭 ❌
   - AI가 의미 비교 ✅
   - 예: "요구사항 명확화" ≈ "사전에 상세 스펙 작성" → 중복

2. **중요도 필터링**:
   - 높음: 실제 버그 발생, 비용 절감, 재사용 가능
   - 낮음: 일회성, 상식, 프로젝트 특수 → 스킵

**교훈**:
- ✅ 자동화할 때 품질 관리 필수
- ✅ 중복/중요도 체크로 노이즈 방지

---

### 4. Git push 에러 처리

**시나리오**:
- 새 프로젝트에서 `/save-progress` 실행
- 원격 저장소가 아직 없음
- `git push origin master` 실패

**해결**:
```
GitHub 저장소가 없습니다. 생성할까요?
- y: gh repo create 자동 실행
- n: 로컬 커밋만 완료
```

**교훈**:
- ✅ 에지 케이스 미리 고려
- ✅ 사용자에게 선택권 제공
- ✅ 실패 시 명확한 안내
